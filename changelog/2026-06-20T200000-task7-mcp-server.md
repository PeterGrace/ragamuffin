# Task 7 — MCP server over stdio

Date: 2026-06-20

## Summary

Replaced the temporary `src/mcp.rs` shim with a real Model Context Protocol
(MCP) server built on `rmcp` 1.7.0. The server exposes two tools over stdio so
LLM coding harnesses (Claude Code, opencode, etc.) can read from and write to
ragamuffin's semantic memory.

## Tools

- `search_memory(query, k=4)` — semantic recall; returns a numbered, cited list
  of the most similar saved notes with their `source` and similarity `score`.
- `save_memory(text, source="chat")` — saves a self-contained note; re-saving
  identical text updates the single existing entry rather than duplicating.

## Implementation notes

- `MemoryServer` holds the `Rag` behind `Arc<Mutex<Rag>>` so the harness may
  issue concurrent tool calls while writes stay serialized.
- Tool registration uses the `#[tool_router]` / `#[tool]` / `#[tool_handler]`
  macro trio; the server is served via `.serve(stdio()).await?` then
  `.waiting().await?`.
- `get_info` advertises `tools` capability and supplies usage instructions to
  the host model.

## Deviations from the planned code

1. `ServerInfo` (alias for `InitializeResult`) is `#[non_exhaustive]` in rmcp
   1.7, so a struct literal with `..Default::default()` does not compile
   (E0639). Construction now starts from `ServerInfo::default()` and mutates the
   `instructions` and `capabilities` fields.
2. `#[tool_handler]` was changed to `#[tool_handler(router = self.tool_router)]`
   so the generated `call_tool` reads the `tool_router` field, eliminating an
   otherwise-spurious `dead_code` warning under `-D warnings`. (The default
   macro form calls `Self::tool_router()` and never reads the field.)
